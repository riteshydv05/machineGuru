# Monitoring Guide

## Overview

MachineGuru exposes comprehensive monitoring data through Prometheus metrics, a stats endpoint, and structured logging. This guide covers all monitoring capabilities and how to set up production-grade observability.

---

## 1. Metrics Endpoint

The backend exposes Prometheus-formatted metrics at `/metrics`:

```bash
curl http://localhost:8000/metrics
```

### Application Metrics

| Metric | Type | Labels | Description |
|--------|------|--------|-------------|
| `machineguru_query_total` | Counter | `status` (ok, error) | Total queries processed |
| `machineguru_query_duration_seconds` | Histogram | — | Query latency buckets |
| `machineguru_query_tokens_total` | Counter | — | Tokens generated by LLM |
| `machineguru_query_cache_hits_total` | Counter | — | Query cache hits |
| `machineguru_query_cache_misses_total` | Counter | — | Query cache misses |
| `machineguru_ingestion_total` | Counter | `status` (ok, error) | Documents ingested |
| `machineguru_ingestion_duration_seconds` | Histogram | — | Ingestion time buckets |
| `machineguru_ingestion_chunks_total` | Counter | — | Total chunks created |
| `machineguru_embedding_duration_seconds` | Histogram | — | Embedding time buckets |
| `machineguru_embedding_cache_hits_total` | Counter | — | Embedding cache hits |

### HTTP Metrics

| Metric | Type | Labels | Description |
|--------|------|--------|-------------|
| `machineguru_request_total` | Counter | `method`, `endpoint`, `status` | HTTP requests by endpoint |
| `machineguru_request_duration_seconds` | Histogram | `method`, `endpoint` | Request latency by endpoint |
| `machineguru_active_requests` | Gauge | — | Currently active requests |

### System Metrics

| Metric | Type | Labels | Description |
|--------|------|--------|-------------|
| `machineguru_memory_mb` | Gauge | `type` (rss) | Process RSS memory |
| `machineguru_gpu_memory_mb` | Gauge | — | GPU memory usage |
| `machineguru_gpu_util_percent` | Gauge | — | GPU utilization |
| `machineguru_concurrent_llm` | Gauge | — | Concurrent LLM requests |
| `machineguru_concurrent_embeddings` | Gauge | — | Concurrent embedding requests |
| `machineguru_concurrent_qdrant` | Gauge | — | Concurrent Qdrant requests |
| `machineguru_model_loaded` | Gauge | `model_name` | Model loaded state (0/1) |

## 2. Stats Endpoint

The `/api/v1/stats` endpoint provides a human-readable snapshot:

```bash
curl http://localhost:8000/api/v1/stats | python3 -m json.tool
```

Response includes:
- Memory (RSS MB)
- CPU usage (%)
- GPU info (memory, utilization, availability)
- Model status (loaded, idle time)
- Cache stats (size, capacity, hit rate)
- Concurrency stats (active, queued, completed per limiter)

## 3. Prometheus Setup

### Configuration

Pre-configured in `deploy/prometheus/prometheus.yml`:

```yaml
scrape_configs:
  - job_name: "machine-guru-backend"
    static_configs:
      - targets: ["backend:8000"]
    metrics_path: "/metrics"

  - job_name: "qdrant"
    static_configs:
      - targets: ["qdrant:6333"]
    metrics_path: "/metrics"

  - job_name: "ollama"
    static_configs:
      - targets: ["ollama:11434"]
    metrics_path: "/api/metrics"
```

### Alerting Rules

Configured in `deploy/prometheus/alerts.yml`:

| Alert | Severity | Expression | For |
|-------|----------|------------|-----|
| BackendDown | critical | `up{job="machine-guru-backend"} == 0` | 30s |
| HighErrorRate | warning | `rate(5xx[5m]) > 0.05` | 2m |
| HighMemoryUsage | warning | `memory_mb > 3072` | 5m |
| SlowQueries | warning | `P95 query > 10s` | 5m |
| LowCacheHitRate | info | `cache hit rate < 10%` | 10m |
| QdrantDown | critical | `up{job="qdrant"} == 0` | 30s |
| HighGPUUsage | warning | `GPU util > 90%` | 5m |

## 5. Health Checks

### Docker Healthchecks

All services have built-in Docker healthchecks that report status via `docker ps`:

```bash
docker ps --format "table {{.Names}}\t{{.Status}}"
```

### External Health Check

Use the health check script for external monitoring:

```bash
# Check all services
./scripts/healthcheck.sh

# Check single service
./scripts/healthcheck.sh backend
```

### API Health Endpoint

```bash
curl http://localhost:8000/api/v1/health
```

## 6. Logging

### Log Format

With `JSON_LOGGING=true`:

```json
{"timestamp": "2026-01-15T10:30:00.123Z", "level": "INFO", "logger": "use_cases.query", "function": "execute", "line": 91, "message": "Query received | text='What are the torque specifications?' top_k=5", "request_id": "abc-123-def"}
```

With console format (default):

```
2026-01-15 10:30:00.123 | INFO    | use_cases.query:execute:91 | Query received | text='What are the torque specifications?' top_k=5
```

### Log Events

Key log events for monitoring:

| Event | Level | Description |
|-------|-------|-------------|
| Application startup | INFO | Version, debug mode |
| Application shutdown | INFO | Models unloaded, uptime |
| Query received | INFO | Query text (truncated), top_k |
| Query cache hit | INFO | Cache key |
| Slow request | WARNING | Request > 1 second |
| Rate limit exceeded | WARNING | Client IP, path |
| Handled exception | WARNING | Error code, status, message |
| Unhandled exception | ERROR | Error ID, stack trace |
| Memory critical | WARNING | Current usage, budget |
| Memory pressure | INFO | Idle models unloaded |
| Ingestion complete | INFO | File stats, timing |
| SIGTERM received | WARNING | Graceful shutdown |

## 7. Performance Tuning

### Cache Optimization

Monitor cache hit rates via `/api/v1/stats`:

```json
{
  "caches": {
    "embedding": {
      "size": 128,
      "capacity": 1024,
      "hits": 95,
      "misses": 200,
      "hit_rate": 0.322
    },
    "query": {
      "size": 15,
      "capacity": 128,
      "hits": 10,
      "misses": 50,
      "hit_rate": 0.167
    }
  }
}
```

Targets:
- Embedding cache hit rate: >50%
- Query cache hit rate: >20% (depends on query diversity)

### Concurrency Tuning

Adjust via `.env`:

```env
MAX_CONCURRENT_LLM=2        # Reduce if OOM
MAX_CONCURRENT_EMBEDDING=1  # Single GPU
```

Monitor concurrency via `concurrent_*` metrics — if any hits the cap consistently, consider increasing.

## 8. Uptime Monitoring

### Docker Restart Policy

All services use `restart: unless-stopped`. Containers restart automatically on crash.

### Watchdog (Optional)

For additional resilience, run the watchdog script:

```bash
./scripts/watchdog.sh --interval 30 --alert \
  --webhook-url https://hooks.slack.com/services/...
```

This checks all services every 30 seconds and sends alerts on failure.

## 9. Key Grafana Queries

Query Latency:
```promql
histogram_quantile(0.95, rate(machineguru_query_duration_seconds_bucket[5m]))
```

Error Rate:
```promql
sum(rate(machineguru_request_total{status="5xx"}[5m])) / sum(rate(machineguru_request_total[5m]))
```

Cache Hit Rate:
```promql
rate(machineguru_query_cache_hits_total[5m]) / (rate(machineguru_query_cache_hits_total[5m]) + rate(machineguru_query_cache_misses_total[5m]))
```

Memory Usage:
```promql
machineguru_memory_mb{type="rss"}
```

GPU Utilization:
```promql
machineguru_gpu_util_percent
```

Throughput:
```promql
rate(machineguru_query_total[5m])
```
